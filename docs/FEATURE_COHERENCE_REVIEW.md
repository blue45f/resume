# 기능 연계 시나리오 + 개연성 전수 검토

작성: 2026-04-18
대상: 전체 피처 세트, 4가지 사용자 여정

## 1. 유저 여정별 검토

### J1. 구직자 (personal) 완주 여정

```
회원가입 → 이력서 작성 → 공개 설정 → 채용 공고 탐색
      → 지원 → 면접 준비 → 모의면접 → 코칭 신청 → 합격
```

| 단계           | 구현                                      | 구멍                                                                 |
| -------------- | ----------------------------------------- | -------------------------------------------------------------------- |
| 회원가입       | ✅ 이메일/OAuth + PIPA 동의               | —                                                                    |
| 이력서 작성    | ✅ Tiptap + 자동저장 + AI 피드백 + JD매칭 | AI 변환에 실패하면 사용자가 원인 파악 어려움 (에러 메시지 개선 필요) |
| 공개 설정      | ✅ public/private/link-only + 슬러그      | link-only 사용시 앱 내에서 공유 UX 불명확                            |
| 채용 공고 탐색 | ✅ 내부 + 외부 큐레이션                   | 외부 공고 클릭 시 외부 사이트 이탈 추적 없음                         |
| 지원           | ✅ JobApplication CRUD + 상태 관리        | 지원 이후 채용 공고 변경 시 추적 안됨                                |
| 면접 준비      | ✅ 질문 풀 + AI 생성 + 스터디 그룹        | 답변 연습 후 피드백 루프 부재                                        |
| 모의 면접      | ✅ 카메라 녹화 + 타이머                   | 녹화본 저장/공유 UX 없음                                             |
| 코칭 신청      | ✅ 예약/결제(stub) + 리뷰                 | 실제 결제 연동 안됨 (PaymentPage stub)                               |
| 합격           | ⚠️ 이력서에 'accepted' 마킹 없음          | 성공 스토리 커뮤니티 기여 유도 없음                                  |

### J2. 리쿠르터 (recruiter) 여정

```
전환 → 인재 탐색 → 북마크 → 스카우트 → 지원자 파이프라인 관리
```

- **전환 UX 부재** (큰 구멍): SettingsPage에 "리쿠르터 전환" UI 없음. 현재는 DB 직접 수정만 가능. → `docs/USER_TYPE_MULTI_ROLE_PLAN.md` Phase 5 필요
- 나머지 단계는 구현됨

### J3. 기업 (company) 여정

- **구현 거의 없음**: userType='company'는 존재하나 팀 초대/기업 브랜딩 페이지/통합 대시보드 미구현
- 현재 company는 recruiter의 상위 alias처럼 동작 (사실상 동일 권한)

### J4. 코치 (coach) 여정

- **구현 양호**: CoachProfile 등록 + 세션 예약 + 리뷰 + 수수료 15% 정산
- **구멍**: 코치 전환 자격 심사 없음 (누구나 즉시 코치 프로필 생성 → 품질 관리 불가)
- **구멍**: 지급 정산은 계산만 되고 실제 지급 플로우 없음

## 2. 역할간 상호작용 매트릭스

| 주체 → 대상 | personal            | recruiter        | company      | coach        | admin   |
| ----------- | ------------------- | ---------------- | ------------ | ------------ | ------- |
| personal →  | 팔로우/쪽지 ✅      | 스카우트 수신 ✅ | -            | 코칭 예약 ✅ | 신고 ✅ |
| recruiter → | 스카우트 발송 ✅    | 쪽지 ✅          | -            | -            | -       |
| coach →     | 세션 수락/수수료 ✅ | -                | -            | 쪽지 ✅      | -       |
| admin →     | 신고 처리/차단 ✅   | 신고 처리 ✅     | 권한 관리 ✅ | 신고 처리 ✅ | -       |

**누락**:

- company ↔ recruiter 팀 개념 부재 (회사가 리쿠르터들을 묶어서 관리)
- coach ↔ company: 기업이 코치를 고용해 팀 전체 코칭하는 B2B 플로우 없음

## 3. 데이터 의존성 이슈

### 3.1 관계 누락

- `CoachingSession` ↔ `Resume`: 코치가 예약 전 피코칭인 이력서를 볼 권한 없음 (JobApplication은 있음)
- `StudyGroup` ↔ `JobPost`: jobPostId는 있으나 그룹 페이지에서 원본 공고 조회 UI 없음
- `CommunityPost` ↔ `Resume`: 커뮤니티 "이력서 피드백" 카테고리에서 첨부 이력서 렌더 없음

### 3.2 상태 일관성

- 이력서 삭제 시 Cascade는 대부분 설정됨 (Comment/Version/Tag/Attachment SetNull or Cascade)
- **유저 삭제 Cascade**: CoachingSession → coach User Cascade 걸려있으나 client Cascade 없음 → 유저 탈퇴 시 상대방 세션 정보 유실
- **Bookmark**: Cascade OK

### 3.3 알림 트리거 누락

| 이벤트               | 알림 생성       | 상태                                                                 |
| -------------------- | --------------- | -------------------------------------------------------------------- |
| 이력서 열람          | ✅ 1시간 쿨타임 | OK                                                                   |
| 스카우트 수신        | ✅              | OK                                                                   |
| 지원 상태 변경       | ❌              | 누락 — recruiter가 파이프라인에서 status 바꿔도 지원자에게 알림 안감 |
| 코칭 예약 확정       | ⚠️              | 부분 — 일부 상태 전이만                                              |
| 커뮤니티 댓글/좋아요 | ⚠️              | 부분 — 댓글만 있고 좋아요 없음                                       |
| 팔로우               | ✅              | OK                                                                   |

## 5. UX 구멍

### 5.1 온보딩

- 회원가입 후 "첫 이력서 만들기" CTA가 HomePage에 묻혀있음
- userType 선택 시 후속 단계 가이드 부재 (예: company 선택 → 기업 등록 절차 안내 없음)

### 5.2 빈 상태

- 대부분 페이지에 EmptyState 있으나, 일관성 부족 (일부는 "데이터 없음" 텍스트만)
- 스터디 카페는 새로 추가했으므로 명시적 EmptyState 확보됨

### 5.3 실패 복구

- 이력서 저장 실패 시 localStorage 백업 있음 (useDraft 훅)
- 그 외 폼 (cover-letter 등)은 백업 없음 → 새로고침 시 데이터 유실

### 5.4 피드백 루프

- 사용자가 기능 오류를 보고할 경로는 Feedback 페이지만 존재
- 페이지 내 "여기서 피드백" 유도 버튼 없음

## 6. 결제/정산 현실성

- PaymentPage/PaymentResultPage는 stub — 실제 PG 연동 없음
- CoachingSession.totalPrice/commission/coachEarn 계산만 되고 실제 이체/정산 프로세스 없음
- 유료 기능(Pro/Enterprise)은 PricingPage에 있으나 구독 DB/상태 없음
- **판정**: MVP 수준의 시뮬레이션. 실서비스로 전환 시 PG/에스크로 연동 필수

## 8. 결론

### 완성도

- **MVP 수준 이상**: 핵심 여정(구직자 여정) 대부분 가동
- **Beta 수준 필요**: 결제, 기업 기능, userType 전환
- **Alpha 수준**: 팀/B2B 기능

### 기능 실사용 가능 여부

✅ **가능**: 이력서 작성/공유, 채용 탐색, 커뮤니티, 면접 준비, 코칭 예약(체험)
⚠️ **제한적**: 리쿠르터 기능 (전환 UI 없어 관리자가 직접 DB 수정)
❌ **불가**: 실제 결제, 회사 단위 팀 관리, 코치 심사

### 기획 완결성

- 구직자 관점: **8/10** (온보딩/피드백 루프만 보완하면 9)
- 리쿠르터 관점: **5/10** (전환 UI + 팀 관리가 큰 gap)
- 기업 관점: **3/10** (거의 미구현)
- 코치 관점: **7/10** (심사·지급만 보완)
- 관리자 관점: **7/10** (기본 기능 + 신고 처리 OK, 상세 분석 부족)

**다음 스프린트 권장**: P0 2건(userType 전환 UI + 결제 베타 배지) + P1 2건 (알림 발송 + 지원 연결)
